# ADR-007: Per-Index StreamingRestorer для стриминга аргументов tool_call

**Date:** 2026-06-12  
**Status:** Accepted

## Контекст

Gateway восстанавливает PII placeholder-ы в аргументах `tool_call` перед возвратом клиенту. В не-потоковом пути это тривиально: сериализуем `tool_calls` в JSON, вызываем `context.restore()`, парсим обратно.

В стриминговом пути `function.arguments` приходит как частичная JSON-строка, разбитая на множество SSE-чанков — часто один или несколько символов на событие. Placeholder вроде `[EMAIL_ADDRESS_3f2a1b0c]` может быть доставлен как `[`, `EMAIL`, `_ADDRESS_3`, `f2a1b0c]` в четырёх отдельных событиях. Побайтовый `str.replace()` на каждом чанке его не найдёт.

## Рассмотренные варианты

**A. Буферизовать все tool_call дельты, отдать один консолидированный чанк в конце**  
Накапливать все фрагменты аргументов по `tool_call index` во время стрима. После завершения стрима — запустить `context.restore()` на собранной строке, yield один дельта-чанк с полными восстановленными аргументами.

Просто в реализации. Но: все чанки аргументов tool_call удерживаются до завершения стрима. Клиент не получает стриминга tool_call — он видит один чанк в самом конце, после того как LLM закончил генерацию. Для текстового контента клиент получает настоящий стриминг; для tool_calls — один поздний выброс.

**B. Побайтовое восстановление с прямой заменой строк**  
Применять `context.restore()` к каждому дельта-чанку по мере поступления. Работает только если весь placeholder приходит в одном чанке. Молча ломается когда placeholder разбит — клиент получает сырой placeholder вместо оригинального значения.

Неприемлемо для продакшна: корректность зависит от поведения токенизатора LLM, которое не гарантировано.

**C. Per-index StreamingRestorer — тот же look-ahead буфер, что для текста (выбрано)**  
Создать один экземпляр `StreamingRestorer` на каждый `tool_call index`. По мере поступления каждого фрагмента аргументов — передавать его в соответствующий restorer. Restorer отдаёт безопасные байты немедленно и буферизует только когда обнаруживает начало потенциального placeholder-а (`[`). Когда полный placeholder найден — оригинальное значение отдаётся одним чанком. Когда буфер не может совпасть ни с одним зарегистрированным placeholder-ом — `[` сбрасывается немедленно.

## Решение

**C. Per-index StreamingRestorer.**

Вариант A был реализован первым, а затем заменён. Подход с консолидированным чанком работал корректно, но ломал стриминг tool_call при наличии PII — что нивелировало смысл стриминга.

## Обоснование

`StreamingRestorer` (v3.3) уже решает идентичную проблему для текстового контента. Применение того же механизма на каждый `tool_call index` повторно использует проверенную реализацию без новой логики. Максимальная задержка ограничена длиной самого длинного зарегистрированного placeholder-а (~26 символов) — незаметно для клиента.

Ключевой инсайт, который делает вариант C корректным: OpenAI streaming клиенты в любом случае должны буферизовать фрагменты `function.arguments` сами перед парсингом JSON. Задержка в один фрагмент при обнаружении placeholder-а на практике неощутима.

Вариант A деградирует стриминг до пакетной доставки для любого запроса с PII. Для коротких tool_calls это приемлемо; для tool_calls с большими наборами аргументов вводит заметную задержку. Деградация также невидима — клиент не получает сигнала о том, что стриминг приостановлен.

## Последствия

- Фрагменты аргументов доставляются клиенту в реальном времени; задержка ограничена длиной placeholder-а
- Чанки могут содержать пустые строки `arguments` в окне буферизации — клиенты уже корректно обрабатывают пустые фрагменты аргументов по спецификации OpenAI streaming
- `finalize()` вызывается на каждом per-index restorer-е после стрим-цикла для сброса оставшегося буфера (например, `[` в конце потока)
- Литеральный `[` в аргументах tool_call вызывает задержку в одно событие; отдаётся немедленно когда следующий символ показывает, что буфер не совпадает ни с одним зарегистрированным placeholder-ом
- Риск ложного срабатывания (LLM генерирует строку, идентичную зарегистрированному placeholder-у) пренебрежимо мал: placeholder-ы используют 4-байтовый случайный hex-суффикс (`secrets.token_hex(4)`), что даёт 2³² возможных значений на тип сущности
