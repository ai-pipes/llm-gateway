# ADR-006: Прозрачный проброс инструментов — схемы на стороне клиента

**Date:** 2026-06-12  
**Status:** Accepted

## Контекст

Gateway проксирует chat-запросы к upstream LLM. Многие клиенты — MCP-адаптеры, agent frameworks, coding assistants — используют OpenAI `tools` API. Нужно было решить, как gateway обрабатывает схемы инструментов: кто их определяет, где они хранятся и что gateway делает с `tool_calls` в ответах.

## Рассмотренные варианты

**A. Реестр инструментов на стороне gateway**  
Gateway хранит реестр известных инструментов (Sentry MCP, внутренние API и т.д.) и инжектирует нужные схемы на основе идентификатора клиента или конфига. Клиент отправляет обычное сообщение; gateway добавляет инструменты перед пересылкой LLM.

**B. Прозрачный проброс — клиент отправляет схемы инструментов (выбрано)**  
Клиент передаёт полный OpenAI-совместимый массив `tools` с каждым запросом. Gateway пересылает его в LLM без анализа. LLM возвращает `tool_calls`; gateway возвращает их клиенту. Клиент выполняет инструмент и отправляет следующий запрос с сообщением `role: "tool"`.

## Решение

**B. Прозрачный проброс.**

## Обоснование

Вариант A превращает gateway в реестр — каждый новый инструмент или MCP-адаптер требует деплоя gateway. Схемы инструментов со временем расходятся с реальными реализациями. Gateway накапливает бизнес-логику, которая принадлежит клиентам, использующим эти инструменты.

Вариант B оставляет gateway тонким прокси. Принцип тот же, что с адаптерами и санитайзерами: gateway определяет точки расширения, клиенты приносят свои реализации. Sentry MCP адаптер — отдельный сервис, управляющий собственными схемами инструментов; он взаимодействует с gateway через стандартный OpenAI-совместимый интерфейс. Никаких изменений в gateway при добавлении нового инструмента или изменении схемы существующего.

Gateway не запускает агентный цикл. Он возвращает `tool_calls` клиенту и ждёт следующего запроса. Это соответствует контракту OpenAI и сохраняет gateway stateless.

## Последствия

- Gateway трактует `tools` как непрозрачный blob — без валидации схем, без бизнес-логики
- `tool_calls` в ответах возвращаются клиенту verbatim после восстановления PII (см. ниже)
- Клиент отвечает за выполнение инструментов и управление циклом диалога
- PII placeholder-ы могут появляться внутри аргументов `tool_call` если LLM их возвращает — gateway восстанавливает оригинальные значения перед возвратом клиенту (стриминг и не-стриминг)
- Проброс `tool_choice` тривиально добавить, но отложен до первой реальной потребности
- Санитизация `function.arguments` на входе вынесена за рамки до появления реальных требований
