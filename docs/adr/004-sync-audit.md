# ADR-004: Синхронная запись audit trail

**Date:** 2026-06-05  
**Status:** Accepted

## Контекст

Audit trail — первый приоритет логирования. Вопрос: записывать синхронно (блокируя ответ клиенту) или асинхронно (fire-and-forget, не влияет на latency)?

## Рассмотренные варианты

**A. Синхронная запись**  
Ответ клиенту отдаётся только после успешной записи аудита. Если запись упала — клиент получает 500.

**B. Асинхронная запись (fire-and-forget)**  
Ответ уходит немедленно, аудит пишется в background. Требует durable backend (Redis Streams, Kafka) для at-least-once гарантии.

## Решение

**A. Синхронная запись.**

## Обоснование

Compliance-требование: если данные переданы клиенту, факт передачи обязан быть зафиксирован. Нарушение этого инварианта создаёт юридические риски в финансовом и медицинском секторах.

Вариант B надёжен только с durable event bus. Без него — at-most-once семантика. С ним — существенное усложнение инфраструктуры (Redis/Kafka как обязательная зависимость для v1).

Latency overhead оправдан: реальные LLM вызовы занимают 500-5000ms. Синхронная запись в stdout добавляет <5ms — менее 1% от общего времени.

## Последствия

- Медленный audit backend напрямую замедляет клиентов — важно для выбора backend в production
- Рекомендуемый backend для высокой нагрузки: stdout → fluentd/logstash (локальный сокет, <1ms)
- Для observability метрик (не compliance) — в v2 добавляется отдельный async `MetricsMiddleware` без этого ограничения
- `BaseAuditBackend` абстрагирует реализацию: stdout (v1), file (v2), postgres (v2)
